12  Selección de Cuencas Prioritarias

Plan de trabajo para identificar cuencas mediante análisis de jerk y validar en terreno

Este capítulo define una planificación de trabajo para seleccionar cuencas prioritarias de validación en terreno a partir del análisis de jerk de la Matriz de Resiliencia Climática Territorial. Su función es conectar el hipercubo espacio-temporal generado por la MRCT con un sistema de consultas reproducible, capaz de identificar cambios abruptos en el tiempo y transformar esas señales en una lista concreta de cuencas visitables.

La planificación está orientada al equipo de análisis espacial y desarrollo. El resultado esperado no es solo un mapa final, sino una base de datos estructurada, un conjunto de consultas documentadas, criterios de selección transparentes y una tabla priorizada de diez cuencas candidatas para mediciones y validación en la Región de Magallanes y de la Antártica Chilena.

El flujo toma como referencia el notebook jupyter-notebook/MRCT_Tutorial_Academico.ipynb, donde se construyen el hipercubo tensorial, el panel RID × Year y las variables de trayectoria temporal: velocidad, aceleración y jerk.

Objetivo operativo

Seleccionar cuencas con cambios abruptos detectados por jerk, cruzarlas con criterios de accesibilidad y cercanía a Punta Arenas, y preparar una lista priorizada para trabajo de terreno con trazabilidad completa entre datos, consulta, año crítico y razón de selección.

12.1 Pregunta de trabajo

La pregunta principal para el equipo es:

¿Qué cuencas muestran cambios abruptos en su trayectoria ecosistémica, en qué años ocurren esos cambios y cuáles de ellas son viables para medición y validación en terreno dentro de las restricciones logísticas del estudio?

Esta pregunta se descompone en cinco preguntas técnicas:

  • ¿cómo almacenar el hipercubo y el panel MRCT para hacer consultas espacio-temporales eficientes?;
  • ¿qué base de datos conviene usar para combinar métricas temporales, geometrías de cuencas, comunas, accesibilidad y cercanía a Punta Arenas?;
  • ¿cómo definir una señal de jerk suficientemente robusta para evitar seleccionar ruido?;
  • ¿cómo priorizar cuencas considerando magnitud del cambio, recurrencia temporal, relevancia territorial y factibilidad de visita?;
  • ¿cómo documentar una tabla final de diez cuencas con año crítico, magnitud del cambio y razón de prioridad?

12.2 Criterio metodológico: por qué jerk

El jerk J mide el cambio de la aceleración de la distancia al dominio:

J_t = A_t - A_{t-1}

Dentro de la MRCT, D indica la posición de una cuenca respecto de su línea base, V describe si se acerca o se aleja de esa referencia, A indica si ese movimiento se intensifica o se amortigua, y J permite detectar cambios abruptos en el régimen dinámico.

Para trabajo de terreno, el jerk es útil porque ayuda a encontrar cuencas donde ocurrió una discontinuidad temporal. Una cuenca puede no tener el peor valor de resiliencia regional, pero sí mostrar un quiebre repentino que merece revisión. Ese quiebre puede estar asociado a cambios climáticos, variación nival, incendios, inundaciones, sequías, cambios de cobertura, eventos extremos, errores de observación o cambios metodológicos en los insumos. Por eso el análisis de jerk no reemplaza la validación: la orienta.

La selección debe evitar una lectura mecánica. Un J alto no prueba degradación, desastre ni transición irreversible. Indica que la trayectoria cambió bruscamente y que conviene revisar el caso con datos complementarios, imágenes, contexto territorial y terreno.

12.3 Flujo general de trabajo

El desarrollo debería organizarse en diez etapas.

12.4 Etapa 1: inventario de insumos

El primer paso es identificar qué datos existen, dónde están almacenados y qué nivel de procesamiento tienen. La selección de cuencas prioritarias no debe comenzar con mapas, sino con una auditoría de datos.

Los insumos mínimos son:

  • panel MRCT RID × Year generado desde el hipercubo;
  • variables estructurales e indicadores base por cuenca y año;
  • métricas de trayectoria: Domain o D, Velocidad, Aceleracion y Jerk;
  • métricas complementarias: Domain_local, SENS, PT, RES, RES_STATE, RES_TYPE y RES_GRAD;
  • geometría de cuencas con campo RID;
  • límites comunales y regionales;
  • capa de caminos, huella urbana o puntos de acceso si está disponible;
  • ubicación del centro urbano de Punta Arenas;
  • atributos de superficie, provincia, comuna y pertenencia territorial.

El equipo debe generar un diccionario de datos con nombre de variable, tipo, unidad, escala temporal, fuente, regla de cálculo, porcentaje de completitud y uso previsto dentro del ranking.

12.5 Etapa 2: selección de base de datos

La base de datos debe permitir consultas sobre tres dimensiones: cuenca, tiempo y espacio. Además, debe ser fácil de reproducir desde el notebook MRCT y suficientemente liviana para análisis exploratorio.

12.5.1 Alternativas evaluadas

12.5.2 Decisión propuesta

La recomendación es usar DuckDB con Parquet/GeoParquet como base analítica local.

Esta opción se justifica por cuatro razones. Primero, el flujo MRCT ya produce una tabla longitudinal RID × Year, que calza naturalmente con Parquet. Segundo, DuckDB permite consultar directamente archivos Parquet sin levantar un servidor. Tercero, SQL facilita dejar consultas reproducibles para el equipo y para el libro digital. Cuarto, la extensión espacial de DuckDB permite cruces con geometrías, distancias, intersecciones y agregaciones territoriales suficientes para esta fase.

PostGIS debe quedar como alternativa de escalamiento si el proyecto necesita edición multiusuario, servicios web, integración con infraestructura institucional o consultas espaciales más pesadas. Para la planificación actual, DuckDB reduce fricción y acelera el paso desde notebook a análisis territorial.

12.7 Etapa 4: preparación del panel consultable

El equipo debe transformar la salida del notebook en una estructura estable de consulta.

Tareas mínimas:

  • exportar el panel MRCT completo a Parquet;
  • exportar las geometrías de cuencas a GeoParquet;
  • normalizar nombres de columnas para evitar ambigüedad entre Domain, D, Domain_local, Jerk y otros nombres equivalentes;
  • asegurar tipos de datos correctos: RID entero, Year entero, métricas numéricas, clases como texto;
  • verificar que cada combinación RID × Year sea única;
  • marcar observaciones con datos incompletos;
  • generar columnas auxiliares como abs_jerk, jerk_sign, jerk_rank_year, jerk_percentile_year y jerk_percentile_global;
  • registrar el último año disponible y la ventana de análisis.

La base debe conservar tanto valores crudos como valores estandarizados. Para selección regional, conviene usar percentiles o z-scores robustos por año, porque el jerk puede tener distribuciones distintas entre años.

12.9 Etapa 6: definición de evento abrupto

No todo valor alto de jerk debe convertirse automáticamente en prioridad de terreno. Se propone clasificar eventos abruptos con criterios combinados.

Un evento candidato ocurre cuando una cuenca-año cumple al menos una de estas condiciones:

  • abs_jerk se ubica sobre el percentil 95 regional del año;
  • abs_jerk se ubica sobre el percentil 99 global de toda la serie;
  • JERK_CLASS corresponde a Intensificación Fuerte o Reversión Fuerte;
  • el evento coincide con aumento de D o deterioro de RES;
  • el evento aparece en una cuenca con alta sensibilidad SENS o alto D_{local}.

Ambos tipos son útiles. La intensificación abrupta puede indicar presión o deterioro emergente. La reversión abrupta puede indicar recuperación, efecto de un evento puntual, cambio hidrológico, variación nival o incluso artefactos de datos que conviene auditar.

12.10 Etapa 7: criterios de priorización

La selección final debe combinar señal MRCT y factibilidad de terreno. Se propone construir un puntaje de prioridad compuesto por cinco dimensiones.

Una formulación simple podría ser:

P_i = 0{,}35 \cdot J^\*_{i} + 0{,}20 \cdot R^\*_{i} + 0{,}15 \cdot E^\*_{i} + 0{,}15 \cdot L^\*_{i} + 0{,}15 \cdot A^\*_{i}

donde J^\* resume magnitud de jerk, R^\* resume relevancia en resiliencia, E^\* resume contraste espacial, L^\* resume factibilidad logística y A^\* representa accesibilidad o cercanía a Punta Arenas. Los pesos son una propuesta inicial y deben quedar parametrizados para sensibilidad.

12.11 Etapa 8: restricción logística Punta Arenas

Como el trabajo de terreno tendrá restricciones de desplazamiento, el sistema debe incluir una consulta específica para cuencas cercanas al centro urbano de Punta Arenas.

Esta consulta debe:

  • identificar cuencas dentro de la comuna de Punta Arenas;
  • identificar cuencas que intersectan un radio operativo alrededor del centro urbano;
  • calcular distancia desde el centroide de la cuenca al centro urbano;
  • calcular distancia a caminos o puntos de acceso cuando la información esté disponible;
  • filtrar cuencas con eventos de jerk relevantes;
  • ordenar primero por factibilidad y luego por magnitud de señal, o producir ambos rankings.

12.13 Etapa 10: fichas por cuenca

Cada una de las diez cuencas seleccionadas debe tener una ficha técnica para terreno. La ficha debe incluir:

  • identificador RID;
  • nombre de cuenca si existe;
  • comuna y provincia;
  • mapa de ubicación regional;
  • mapa de detalle con acceso preliminar;
  • año crítico de jerk;
  • serie temporal de D, V, A, J y RES;
  • indicadores base que más cambiaron alrededor del año crítico;
  • clasificación RES_STATE y RES_TYPE;
  • distancia a Punta Arenas y ruta estimada si existe;
  • observaciones para terreno;
  • hipótesis preliminar a validar;
  • fotografías o puntos sugeridos de observación si se definen posteriormente.

Estas fichas permiten que el equipo de terreno no reciba solo una lista de coordenadas, sino una pregunta de validación por cada cuenca.

12.14 Consultas SQL orientativas

Las consultas finales deben ajustarse al nombre real de tablas y columnas, pero el equipo puede partir con una lógica similar a la siguiente.

-- Eventos de jerk extremos por año
CREATE OR REPLACE TABLE jerk_events AS
SELECT
  RID,
  Year,
  Jerk,
  abs(Jerk) AS abs_jerk,
  CASE
    WHEN Jerk > 0 THEN 'Intensificacion'
    WHEN Jerk < 0 THEN 'Reversion'
    ELSE 'Neutro'
  END AS jerk_sign,
  percent_rank() OVER (
    PARTITION BY Year
    ORDER BY abs(Jerk)
  ) AS jerk_percentile_year
FROM mrct_panel
WHERE Jerk IS NOT NULL;
-- Candidatas regionales con resiliencia y comuna
SELECT
  e.RID,
  e.Year AS critical_year,
  e.Jerk,
  e.abs_jerk,
  e.jerk_percentile_year,
  p.RES,
  p.PT,
  p.Domain,
  p.Velocidad,
  p.Aceleracion,
  p.Domain_local,
  p.SENS,
  bc.commune_name
FROM jerk_events e
JOIN mrct_panel p
  ON e.RID = p.RID AND e.Year = p.Year
LEFT JOIN basin_commune bc
  ON e.RID = bc.RID
WHERE e.jerk_percentile_year >= 0.95
ORDER BY e.abs_jerk DESC;
-- Candidatas cercanas a Punta Arenas
SELECT
  c.*,
  a.distance_punta_arenas_km,
  a.distance_road_km,
  a.access_ring
FROM candidate_events c
LEFT JOIN accessibility a
  ON c.RID = a.RID
WHERE a.access_ring IN ('Cercano', 'Intermedio')
ORDER BY
  c.jerk_percentile_year DESC,
  a.distance_punta_arenas_km ASC;

Estas consultas deben transformarse en scripts versionados para que el ranking pueda regenerarse cuando cambien los datos, el año de análisis o los pesos de priorización.

12.18 Riesgos y controles

12.19 Cierre operativo

La selección de cuencas por jerk debe funcionar como un puente entre análisis computacional y observación territorial. El hipercubo MRCT permite detectar discontinuidades temporales que no son evidentes en una lectura estática de resiliencia. La base de datos y el sistema de consultas convierten esa señal en un procedimiento reproducible. La restricción logística de Punta Arenas asegura que el resultado sea viable para una campaña real.

El producto final para el libro digital será una lectura trazable: desde el panel RID × Year, pasando por eventos de jerk, filtros espaciales y criterios de accesibilidad, hasta una lista de diez cuencas priorizadas con año crítico y razón técnica de selección.
